Overview

What is an Operator?

  • An Operator represents human operational knowledge in software, to reliably manage an application

  • An Operator is a type of Controller which manage Resources

    • Resources: Pod, ConfigMap, Secret, Route, Ingress, …

    • Controllers: Deployment, ReplicaSet, StatefulSet, DaemonSet

  • An Operator can use stock controllers or manage resources itself

Interacting with OpenShift

  • CRDs allow us to extend the Kubernetes/OpenShift API

  • CRDs allow us to create Custom Resources (CR)

  • CRs can be managed just like stock OpenShift objects

    • create, get, describe, delete, …

Operator Framework

  • https://github.com/operator-framework

=== Transcript The Operator Framework consists of three parts - but is all designed to provide a streamlined, uniform way of writing and managing Operators (which again, can be deployed to ANY Kubernetes cluster, not just OpenShift).

  • Operator SDK (Software Development Kit) makes it easy to build or modify Operators for your own organizations - or, if you’re an ISV, for your particular service - without specialized knowledge of the Kubernetes API. Focus more on building what your organization needs, not on the intricacies of Kubernetes.

  • The Operator SDK is not in scope for this training because it covers building new operators.

  • Operator Lifecycle Manager (OLM) manages the lifecycle of all Operators and their dependencies on your Kubernetes cluster. With OLM, you can install applications with a single kubectl command, and you’re not saddled with manually managing permissions or updates.

  • Operator Metering This portion of the framework gives admins insight into usage for specific Operators. Usage metrics are unique to each Operator - metering watches custom resources and collects data, and performs calculations and reports on that data back to the administrator.

Operator Components

Installing an Operator

  1. Create the Custom Resource Definition (cluster-admin)

  2. Create an RBAC Role (cluster-admin or admin)

  3. Create an RBAC Role Binding (for the service account that the operator runs under) (cluster-admin or admin)

  4. Create the Operator (by pointing to a container image that contains the operator code) (cluster-admin or admin)

=== Transcript CRDs are Cluster Resources. Therefore cluster-admin authorization is necessary to deploy/create a new CRD in a cluster. RBAC Roles can be cluster scoped (if the Operator needs to do "cluster-wide" things) or project scoped. Operators can be cluster scoped or namespace scoped (one operator per namespace)

Operator Components

Custom Resource Definitions
apiVersion: apiextensions.k8s.io/v1beta1
kind: CustomResourceDefinition
metadata:
  name: gogs.gpte.opentlc.com
spec:
  group: gpte.opentlc.com
  names:
    kind: Gogs
    listKind: GogsList
    plural: gogses
    singular: gogs
  scope: Namespaced
  version: v1alpha1

=== Transcript This Custom Resource Definition defines an object called Gogs. It also specifies that the plural of Gogs is Gogses. After creating this CRD the Kubernetes/OpenShift API understands commands like oc get gogses or oc describe gogs mygogs.

Operator Definition
apiVersion: apps/v1
kind: Deployment
metadata:
  name: gogs-operator
spec:
  replicas: 1
  selector:
    matchLabels:
      name: gogs-operator
  template:
    metadata:
      labels:
        name: gogs-operator
    spec:
      serviceAccountName: gogs-operator
      containers:
        - name: gogs-operator
          image: quay.io/wkulhanek/gogs-operator:v0.0.1
          ports:
          - containerPort: 60000
            name: metrics
          imagePullPolicy: Always
          env:
            - name: WATCH_NAMESPACE
              valueFrom:
                fieldRef:
                  fieldPath: metadata.namespace
            - name: POD_NAME
              valueFrom:
                fieldRef:
                  fieldPath: metadata.name
            - name: OPERATOR_NAME
              value: "gogs-operator"

Ansible Operator

Overview

  • Provides a Base Operator Image

  • Execute a Playbook or a Role

  • Only Creates Objects in OpenShift, no deletion

  • Adds ownerReferences to all created objects

  • Keeps running the Playbook/Role to ensure Objects are there

    • Will recreate changed/deleted objects

  • User’s Guide:

Operator Lifecycle Manager

Overview

  • The Operator Lifecycle Manager is the following for Operators

    • Installation (InstallPlan := Package Manager)

    • Definition (ClusterServiceVersion := Package)

    • Resolution (CatalogSource := Repository)

    • Upgrading (Subscription)

  • Implemented itself as a set of Operators

    • OLM Operator, Catalog Operator, Package Server

=== Transcript All objects (InstallPlan, ClusterServiceVersion, …) are CRDs.

Operator Lifecycle Manager

Deploying an Operator via OLM

  • Create Subscription from the Catalog (e.g. etcd)

  • Subscription creates InstallPlan

  • InstallPlan creates Service Account, Role, Role Binding, ClusterServiceVersion and ultimately the Operator

  • Select a ClusterServiceVersion and create an Instance (of e.g. etcd Cluster)

    • This creates the Custom Resource defining the parameters of the workload

    • When the Custom Resource has been created the Operator gets an event to deploy whatever is defined as the custom resource
