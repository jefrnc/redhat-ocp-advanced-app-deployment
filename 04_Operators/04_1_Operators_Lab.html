Operators Lab

Operators are a way to create custom controllers in OpenShift and Kubernetes. They use the concept of Custom Resource Definitions to extend the Kubernetes API. The Operator then watches for custom resource objects to be created - and when it sees a custom resource being created it creates the application based on the parameters in the custom resource object.

Operators are often written in the Go programming language (just like Kubernetes and OpenShift). This means that the Kubernetes API is used to create and monitor objects in the cluster directly.

Since this is not a very developer friendly experience Red Hat has provided a software development kit (SDK) that significantly simplifies the creation of Operators. The SDK handles all the core logic while the developer only needs to fill in the actual business logic.

Red Hat also provides other capabilities in the SDK to create operators from either Ansible Roles or Playbooks as well as from Helm Charts.

In this lab you are using the Operator SDK to create an Operator using two Ansible Roles and a Playbook that deploy a PostgreSQL database and a Gogs server that uses that database. Unlike using templates, this operator can fully configure the Gogs Server since it can query the Route that has been created before adding it to the Gogs configuration.

See the Ansible Operator Documentation for more information.

See the Helm Operator Kit for more information about Helm Charts. Note that Helm Charts need to satisfy the same requirements as regular OpenShift applications (e.g. Pods do not run as root) in order to run successfully on OpenShift.

Goals

  • Create an Operator from Ansible Roles using the Ansible Operator SDK

  • Deploy the Operator to OpenShift

  • Create a Gogs Server using a Custom Resource

  • Create a second Gogs server

  • Examine the created objects

  • Delete extra Gogs server

Create the Gogs Operator

The first step to building an Operator is to install the Operator SDK into your Client VM.

  1. Download the Operator SDK binary into /usr/local/bin:

    This lab uses version v0.16.0 because the Operator SDK changes quite frequently and you need to use a well-defined snapshot of the repository to guarantee a smooth lab.

    sudo wget https://github.com/operator-framework/operator-sdk/releases/download/v0.16.0/operator-sdk-v0.16.0-x86_64-linux-gnu -O /usr/local/bin/operator-sdk
sudo chmod +x /usr/local/bin/operator-sdk

  2. Now your Operator SDK is ready to be used.

  3. There is an existing repository that includes the Ansible Roles to create a PostgreSQL database and a Gogs Server running on OpenShift. The repository also contains an Ansible playbook that calls these roles to create a Gogs Server backed by a PostgreSQL database in a project on OpenShift.

    You will now use the Ansible Operator to convert these roles into an Operator that can create Gogs installations by simple requesting a Custom Gogs Resource.

  4. Clone the repository with the roles.

    cd $HOME
git clone https://github.com/redhat-gpte-devopsautomation/ansible-operator-roles
cd ansible-operator-roles
git checkout v0.16.0
cd $HOME

  5. Examine the Playbook:

    cat $HOME/ansible-operator-roles/playbooks/gogs.yaml

  6. You can also examine the two roles (and the default values for their different variables):

    cat $HOME/ansible-operator-roles/roles/postgresql-ocp/tasks/main.yml
cat $HOME/ansible-operator-roles/roles/postgresql-ocp/defaults/main.yml
cat $HOME/ansible-operator-roles/roles/gogs-ocp/tasks/main.yml
cat $HOME/ansible-operator-roles/roles/gogs-ocp/defaults/main.yml

  7. From your home directory, use the operator-sdk binary to create a new Ansible Operator skeleton (you can use the Ansible Operator User’s Guide for reference). Use the following values:

    • gogs-operator as the operator name

    • gpte.opentlc.com/v1alpha1 as the API version

    • Gogs as the Kind

      If you don’t use these values, your Operator will not work in this shared cluster.
      Make sure to pick Ansible as the operator type. Also, make sure to include the option to use a playbook and not just roles.
      If you made a mistake, you can delete the directory (gogs-operator) and try again.

  8. The operator-sdk creates a skeleton playbook and a skeleton roles directory. But, you already have a playbook and roles from the repository you cloned earlier. Switch into the directory and replace the playbook and roles directory with the ones you downloaded.

    cd $HOME/gogs-operator
rm -rf roles playbook.yml
mkdir roles
cp -R $HOME/ansible-operator-roles/roles/postgresql-ocp ./roles
cp -R $HOME/ansible-operator-roles/roles/gogs-ocp ./roles
cp $HOME/ansible-operator-roles/playbooks/gogs.yaml ./playbook.yml

  9. Examine the file watches.yaml. This is the main configuration for the Ansible Operator.

    cat ./watches.yaml
    Sample Output
    ---
- version: v1alpha1
  group: gpte.opentlc.com
  kind: Gogs
  playbook: playbook.yml

    You will notice that the group, version and kind came from the command you used to create the skeleton. The Playbook uses the playbook.yml filename. You will see how the file ends up in the location /opt/ansible in the next step.

  10. Next examine the Dockerfile in the build directory:

    cat build/Dockerfile
    Sample Output
    FROM quay.io/operator-framework/ansible-operator:v0.16.0

COPY requirements.yml ${HOME}/requirements.yml
RUN ansible-galaxy collection install -r ${HOME}/requirements.yml \
 && chmod -R ug+rwx ${HOME}/.ansible

COPY watches.yaml ${HOME}/watches.yaml

COPY roles/ ${HOME}/roles/
COPY playbook.yml ${HOME}/playbook.yml

    Notice the source image for the container image that is being built. Also notice that all the Dockerfile does is to copy the roles, playbook, watches, and requirements files into the container image. The requirements.yaml file is then used during the container images build to install a couple of Ansible Collections.

    The ansible-operator container image already contains the logic that an Operator needs and just uses the playbook to set up the application.

    The playbook needs to be idempotent (meaning it must not fail when run a second time) because the operator keeps running the playbook periodically to ensure all necessary objects are in place.

    While the roles in this example support removing the applications (by setting state=absent) this is not necessary for roles in Ansible Operators. When the Ansible Operator creates Kubernetes Objects it injects ownerReference fields into all created objects. Therefore deleting the CR (Gogs in our example) will delete all associated objects.

  11. In order to build and push the container image to a registry you will need a user id on a publicly accessible container image registry. For the purposes of this lab it is recommended that you create an account at https://quay.io if you do not already have one.

    The create account link is a bit hidden. Click on Sign In in the top right corner to see the login screen - which also has the link to Create Account.

    Once you have created your Quay account make sure to also create a public repository in Quay called gogs-operator. This will be where you will push your container image. Ask the instructor if you need any help with this.

    Make sure to select the public radio button on the Create New Repository page.

  12. Once you have your Quay.io user ID and repository created, you will need to log into Quay for the subsequent podman push command to succeed.

    podman login -u <your user> quay.io

  13. Now use the Operator SDK to build the container image and tag it as quay.io/<your quay id>/gogs-operator:v0.0.1.

    By default, the operator-sdk will try to use Docker to build the image. Since you are using podman, pass in the --image-builder podman option to the operator-sdk build command. See the operator-sdk help for more details.

  14. Now push the container image to Quay:

    podman push quay.io/${QUAY_ID}/gogs-operator:v0.0.1

  15. The final step is to modify the Operator deployment definition (deploy/operator.yaml) to use the image that you just pushed.

Deploy the Gogs Operator

Overview

Now that everything has been built you can deploy the Gogs Operator. In this lab, the cluster already has the Custom Resource Definition deployed because cluster-admin permissions are necessary to deploy a new CRD. As a matter of fact, as a regular user you can’t even list the available CRDs.

If you had cluster admin privileges, you could run the command to see the CRDs:

oc get crd
Sample Output
NAME                                          CREATED AT
[...]
gogs.gpte.opentlc.com                         2018-11-12T17:35:07Z
[...]

Notice the gogs.gpte.opentlc.com Custom Resource Definition.

Deploy the Operator

Operators can be deployed on a cluster-wide basis or on a namespace/project basis. In this lab you are deploying your operator in your own project. You will use this Gogs instance in the following labs as well.

  1. Create the operator project:

    oc new-project ${GUID}-gogs --display-name="Gogs"

  2. Now deploy one by one the artifacts in the deploy subdirectory.

  3. The first one to deploy is the service_account that our Operator will use.

    oc apply -f $HOME/gogs-operator/deploy/service_account.yaml

  4. Next are the Role and RoleBinding that grant the correct permissions to the Operator. The Operator SDK will create a skeleton Role file for you. However, depending on what the Operator needs to create or manipulate, it will most likely be necessary to customize the Role. Using the following requirements, edit the $HOME/gogs-operator/deploy/role.yaml file:

    • The Gogs Operator will need to create and manage the routes resource, which is part of the route.openshift.io API group. You will need to add this. Give it the same verbs as the "" API group.

    • Replace the "*" resource in the gpte.opentlc.com API Group with gogs and gogs/status

    • Add serviceaccounts resource under apiGroup "".

    • Remove any access to the monitoring.coreos.com API group. You will not have access.

  5. Create the Role and RoleBinding (granting the permissions in the role to the service account that the Operator uses):

    oc apply -f $HOME/gogs-operator/deploy/role.yaml
oc apply -f $HOME/gogs-operator/deploy/role_binding.yaml

  6. Now that all of the prerequisites are in place to launch the Gogs Operator, proceed with creating the Deployment that points to your Operator image.

    oc apply -f $HOME/gogs-operator/deploy/operator.yaml

  7. Wait until the Operator is running (the first time may take a while because the container image needs to be pulled from Quay - and this image is rather big). Using the Operator to create a Gogs Server

To use an Operator to deploy an application it is necessary to create a Custom Resource - which is a Kubernetes Object. When the custom resource gets created in a project that the Operator is watching the Operator reacts and creates the objects that make up that application based on the configuration options in the custom resource.

The Gogs Operator understands the following Configuration Options:

Name

Default

Required

postgresqlVolumeSize

1Gi

No

gogsVolumeSize

1Gi

No

gogsSsl

False

No

In Kubernetes, the convention for variable names is to use CamelCase while in Ansible it is snake_case. Therefore the Custom Resource needs the variable names to be in CamelCase - while the Ansible Playbooks and Roles expect them in Snake Case. The Ansible Operator base image handles this translation automatically.

The generated file deploy/crds/gpte.opentlc.com_gogs_crd.yaml includes a skeleton custom resource. Since the operator SDK generator can not know which variables the playbooks in the Ansible Operator understand, you will need to customize the CR.

When writing an Operator you will need to consider that there may be more than one instance of Operator created objects in a project. In the Gogs example that means there may be more than one PostgreSQL Database and more than one Gogs server.

Any objects that get created need to have a unique name - usually this is achieved by using the name of the custom resource as a prefix to any created object. The roles you are using are built to satisfy these requirements.

  1. Create a Custom Resource to deploy a Gogs Server in your project.

    echo "apiVersion: gpte.opentlc.com/v1alpha1
kind: Gogs
metadata:
  name: gogs
spec:
  postgresqlVolumeSize: 4Gi
  gogsVolumeSize: 4Gi
  gogsSsl: True" > $HOME/gogs-operator/gogs.yaml

  2. Now create the Gogs server:

    oc apply -f $HOME/gogs-operator/gogs.yaml

Deleting a Gogs Server

Deleting an object created by an Operator is as easy as deleting the custom resource. Operators that follow best practices set an ownerReference field in all created objects to enable cascading deletions. The Ansible Operator watches for Kubernetes API calls to create resources and adds the ownerReference field to every created resource after it’s been successfully created.

  1. Examine the Owner Reference of the created pods. First retrieve a list of pods.

    oc get deployments
    Sample Output
    NAME                           READY   UP-TO-DATE   AVAILABLE   AGE
gogs-another-gogs              1/1     1            1           3m32s
gogs-gogs                      1/1     1            1           7m2s
gogs-operator                  1/1     1            1           11m
postgresql-gogs-another-gogs   1/1     1            1           4m29s
postgresql-gogs-gogs           1/1     1            1           9m54s

  2. Get the YAML output of the first PostgreSQL Deployment. Note the ownerReference points to a Gogs that has the name gogs. This object gogs of kind Gogs is the custom resource that you created.

    oc get deployment postgresql-gogs-gogs -o yaml
    Sample Output

    oc delete gogs another-gogs
Do not delete the first instance of Gogs.
