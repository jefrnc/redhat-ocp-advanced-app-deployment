Homework Assignment Lab

Goal

  • Assess hands-on proficiency with Red Hat OpenShift Advanced Application Deployment topics

Criteria

  • Expect the assignment to take approximately 8 hours to complete

  • Assignments are an individual effort — complete assignments without collaboration

  • Assignments simulate a challenge typically encountered in a Red Hat consulting engagement

  • Assignments are graded automatically via a grading pipeline

Grading

  • Assignments must pass the automated grading pipeline

Submission

  • All code must be in a private Gitea repository:

    • Gitea Repository. Create a user id that is the same as your OpenTLC user (e.g. wkulhane-redhat.com). Password can be anything you would like it to. Because you need to submit your repository password it is strongly suggested that you pick a password that is different from your OpenTLC User Account.

  • Submit your homework documentation including a link to the private repository (including your OpenTLC User ID and Git repository password) and the successful build number from the Grading pipeline.

  • Instructions for submitting your homework:You submit your homework via one of the following:

    • Red Hat employees: Red Hat Employee Homework Submission Instructions - PDF

    • Partners: Partner Employee Homework Submission Instructions - PDF

Environment

  • The assignment must be submitted using the OpenShift 4.4 Advanced App Deploy Homework environment.

    OpenShift 4.4 Advanced App Deploy Homework Environment

    • You will use a shared cluster to build/submit your homework assignment.

    • As usual, you can request access in the OPENTLC lab portal

    • You need the following two items from the OpenShift 4 Labs catalog:

      • OpenShift 4.4 Adv App Deploy Homework: This item gives you access to the OPENTLC OpenShift 4 Shared Cluster environment. Once you provision this item, you can create projects and you are provided with sufficient quota to complete the homework assignment. You also get access to the homework grading Jenkins.

      • OpenShift 4 Student VM: Use the instructions in the email that you receive from Red Hat to access this VM.

        • The student VM is optional. You can also use the oc client from your laptop if you prefer.

1. Business Use Case

You are a consultant assigned to MitziCom, a telecommunications company that provides hosting and cloud services to a variety of clients, from medium-sized companies to enterprise giants.

You just attended a proof of concept workshop where you co-developed a pipeline to deploy one of MitziCom’s applications, Tasks, into a playground cluster.

MitziCom liked the results of the proof of concept so much that they have asked you to deploy the same pipeline into their production environment - using the shared services that are already available on the cluster. This includes Nexus and SonarQube as well as a Nexus container registry.

The source code for the tasks application can be found at https://github.com/redhat-gpte-devopsautomation/ocp4_app_deploy_homework_template.

The automatic grading pipeline can be found at https://github.com/redhat-gpte-devopsautomation/ocp4_app_deploy_homework_grading/tree/update_44. You do not need to clone this repository - but it may be helpful to understand how the automatic grading works.

2. CI/CD Infrastructure

  • MitziCom has a shared Nexus Artifact Repository deployed in the cluster.

    • The Service for Nexus is homework-nexus.gpte-hw-cicd.svc.cluster.local:8081

    • The public route for Nexus is https://homework-nexus.apps.shared-na4.na4.openshift.opentlc.com

    • The Service for the Nexus Container Registry is https://homework-nexus-registry.gpte-hw-cicd.svc.cluster.local:5000

    • The public route for the Nexus Container Registry is https://homework-nexus-registry.apps.shared-na4.na4.openshift.opentlc.com

    • The repository group is /repository/maven-all-public

  • MitziCom has a Shared SonarQube installation deployed in the Cluster.

    • The SonarQube service is homework-sonarqube.gpte-hw-cicd.svc.cluster.local:9000.

    • The SonarQube public route is http://homework-sonarqube.apps.shared-na4.na4.openshift.opentlc.com

3. POC Requirements

3.1. Jenkins Project Setup

  • You will need to modify the provided shell script to create your personal Jenkins environment.

    • Create a persistent Jenkins in project ${guid}-jenkins.

    • Create a custom agent image (which includes skopeo) to build your pipeline.

    • The agent image must have the name jenkins-agent-appdev in project ${guid}-jenkins.

    • Create a ConfigMap holding the agent pod definition for your custom agent image. Use maven-skopeo-agent as the name and label of the pod template.

    • Create a secret with access credentials to your private repository

    • Create a build config pointing to the pipeline source (Jenkinsfile) in your private repository.

3.2. Pipeline

  • Create a Jenkins pipeline for the Tasks application and store it in the source code repository. The directory openshift-tasks contains a shell Jenkinsfile for you to get started.

  • Make sure that the development pipeline executes the following steps:

    • Build the source code, using the shared Nexus as a Maven proxy cache

    • Execute Unit tests

    • Code coverage tests using the shared SonarQube instance

    • Tag the image with the version from the pom.xml and build number

    • Upload the generated artifact to the shared Nexus artifact repository

    • Upload the tested container image to the shared Nexus container registry

  • Deploy the application into the development project

  • Tags the image as version for production deployment

  • Deploys the application into the production project using a blue-green strategy

    • Automatically executes the blue-green go-live switch

4. Instructions

  1. Clone the shell source code repository (https://github.com/redhat-gpte-devopsautomation/ocp4_app_deploy_homework_template) to your hard drive.

    • Register an account for the homework repository at https://homework-gitea.apps.shared-na4.na4.openshift.opentlc.com.

      • The user id for the account must be the same as your OpenTLC user. The password can (and should) be different.

    • Create a private repository at https://homework-gitea.apps.shared-na4.na4.openshift.opentlc.com with your homework code.

  2. Populate the private repository with the contents of the shell source code repository.

  3. Implement the shell scripts under bin.

    • setup_jenkins.sh: This script needs to do the following in the $GUID-jenkins project:

      • Create a Jenkins instance with persistent storage and sufficient resources

      • Create a build configuration to build the custom Maven agent image to include skopeo

      • Create a ConfigMap with the pod Template definition of your Maven agent image

      • Create a secret holding the access credentials for your private repository. While not ideal you can add your userid and password to this shell script because the shell scripts lives in your private repository and can not be seen by anyone else.

      • Set up a build configuration with a pointer to the pipeline in the source code project using openshift-tasks as the contextDir.

        Use the previously created secret to allow the build configuration to pull the Jenkinsfile from your repository.

        The build configurations requires the following environment variables to be set. These variables are passed into the Shell script from the grading pipeline.

        • GUID: The common GUID for all projects

        • REPO: The URL to the source code repository to retrieve the Jenkinsfile from

        • CLUSTER: The cluster base URL—for example, shared.na.openshift.opentlc.com

    • setup_projects.sh: Does not need to change

      • In order to test outside of the official Jenkins Pipeline the FROM_JENKINS parameter can be set to false to skip the tasks that require cluster-admin privileges

    • reset_prod.sh: Does not need to change

    • delete_projects.sh: Does not need to change

    • setup_dev.sh: Does not need to change

    • setup_prod.sh: Does not need to change

  4. Implement the pipeline for the tasks application keeping in mind that a shared Nexus and Sonarqube is to be used.

  5. Because Nexus is a shared environment make sure that the name of your application image is unique. The shell pipeline defines imageName ="${GUID}-tasks". Make sure to use the variable imageName wherever you are referencing the tasks application container image.

  6. The grading pipeline tests for the contents of the environment variable VERSION which is surfaced on the homepage of the tasks application.

    • Set the environment variable VERSION on the tasks deployment config in the test project to '${devTag} (tasks-dev)'.

    • Set the environment variable VERSION on the blue tasks deployment config in the production project to '${prodTag} (tasks-blue)'

    • Set the environment variable VERSION on the green tasks deployment config in the production project to '${prodTag} (tasks-green)'

  7. Test the pipeline via the pipeline build configuration in your Jenkins project.

    • Once the pipeline succeeds, you are ready to test the environment.

5. Homework Submission

5.1. OPENTLC Grading Jenkins

When you order the OpenShift 4.4 Adv. App Deploy Homework environment from OpenTLC, you also get permission to view the grading Jenkins. This means that you can go to Jenkins and kick off a grading pipeline yourself.

  1. Access the OPENTLC Jenkins http://homework-jenkins-gpte-jenkins.apps.shared-na4.na4.openshift.opentlc.com.

  2. Create Credentials in Jenkins to hold your userid and password for the private repository with your submission.

    • In the left Navigation click Credentials, then System.

    • In the area on the right click on Global credentials (unrestricted).

    • On the left click Add Credentials

    • Use the following values (CAREFUL - YOU CANNOT CHANGE THESE ONCE CREATED)

      • Kind: Username with Password

      • Scope: Global (Jenkins, nodes, items, all child items, etc)

      • Username: Your OpenTLC Username - which should be the same for the private repository with your homework code

      • Password: Your password for the private repository with your homework code

      • ID: Your OpenTLC Username. If this does not match the grading pipeline will not be able to find your credentials

      • Description: "Credentials for <OpenTLC User ID> user"

  3. Once you are certain that the credentials are correct, click OK.

  4. Go back to the Jenkins Homescreen by clicking the Jenkins logo in the top left of the screen.

  5. Run the Grading Pipeline.

    • Click Grade Advanced Application Deployment Homework, select Build with Parameters, fill in your specific values, and build.

Grading Pipeline

The grading pipeline does the following:

  • Checks out your source code repository

  • Ensures all shell scripts in the repository are executable

  • Calls shell scripts to do the following:

    • Create projects

    • Set up Jenkins (for you to implement)

    • Set up the development project

    • Set up the production project

  • Executes the pipeline for the Tasks application for the initial (green) deployment.

  • Test if the correct value is returned

  • Executes the pipelines again to execute the blue deployment

  • Test if the correct value is returned

  • Delete the projects

  • Future: Update the LMS with success or failure of the pipeline

If the pipeline completes successfully you have passed the homework assignment.

5.2. Hints

  • Once your infrastructure is set up correctly, you may set the SETUP variable to false for subsequent runs. This causes the pipeline to skip the initial project setup and call reset_prod.sh instead to reset to a defined state for the two pipeline runs.

  • It is usually a good idea to leave DELETE set to false while you are testing your pipelines.

  • The created projects are annotated with your ${user} - which means they will be deleted when the homework environment gets deprovisioned.
